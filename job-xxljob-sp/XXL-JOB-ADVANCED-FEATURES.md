# XXL-JOB 进阶核心特性指南

在掌握了 XXL-JOB 的“调度与执行器解耦”、“BEAN 模式运行”、“路由策略”以及“分片广播计算”后，掌握以下 5 个高级特性，将极大提升在生产环境下的实战与排错能力。

---

## 1. 生命周期管理：`init` 和 `destroy`

默认的 `@XxlJob` 仅仅包含业务执行逻辑。如果任务在运行前后需要严格的资源分配与释放，可以通过配置绑定生命周期方法。

### 核心场景
*   **Init (初始化)**：建立耗时的 SFTP 连接池、临时打开大型文件流、初始化特殊的上下文配置。
*   **Destroy (销毁)**：任务结束后清理临时文件、关闭数据库游标或特殊网络连接、打扫战场防内存泄漏。

### 代码演示
```java
@XxlJob(value = "demoJobHandler", init = "initMethod", destroy = "destroyMethod")
public void demoJobHandler() throws Exception {
    // 核心业务处理逻辑...
}

public void initMethod() {
    log.info("【任务生命周期】任务开始前，分配重量级资源...");
}

public void destroyMethod() {
    log.info("【任务生命周期】任务跑完了，释放连接，清理临时文件...");
}
```

## 2. 阻塞处理策略 (Block Strategy)

**高频面试题与生产痛点。**
当一个任务的执行时间 **超出了** 它的调度周期（例如：原定1分钟跑一次，但因为数据量突增，该任务跑了 3 分钟还没结束），此时到达第2分钟的调度触发点，Admin 又下发了执行指令。执行器该如何处理这种冲突？

在 Admin 控制台新建/编辑任务时，可以选择以下策略：

1.  **单机串行（默认，最安全）**：执行器收到新指令后，发现老任务还在跑，就把新指令塞到后台队列里排队。等老任务跑完，立马接着按顺序跑下一个。
2.  **丢弃后续调度**：如果发现当前任务还在执行，直接将 Admin 最新发过来的指令**无视并丢弃**。适合具有幂等性或后续调度可以补齐业务逻辑的场景。
3.  **覆盖之前调度**：非常暴力。执行器会直接调用正在运行线程的 `interrupt()` 方法强行中断老任务，并丢弃老任务的数据，强制立马开始执行最新的调度指令。极少使用，除非对任务实时性要求极高。

## 3. 子任务机制 (轻量级工作流)

XXL-JOB 虽然不是像 Camunda 那样重量级的工作流（BPMN）引擎，但它提供了轻量级的任务依赖编排设定。

*   **配置方式**：在 Admin 后台任务配置面板，可以设置 **子任务 ID**（填写需要依赖此任务的其他任务 ID，支持英文逗号分隔拼接多个）。
*   **运行逻辑**：**当父任务执行完全成功后，XXL-JOB Admin 会自动立即触发后续的子任务。**
*   **应用场景**：极度适合简单的链式流水线。
    *   任务 A（ID=10）：拉取银行对账文件。
    *   任务 B（ID=11）：对账文件解析入库。（B 设为 A 的子任务）
    *   任务 C（ID=12）：生成差异报表清单。（C 设为 B 的子任务）

## 4. GLUE 模式（Web IDE 动态脚本）

日常绝大部分业务是 **BEAN 模式**，需要写 Java、打包、部署运行进程。但在运维或者临时性的数据处理场景，XXL-JOB 提供了一项“神器”—— GLUE 模式。

### 特权与优势
*   **语言无关性**：可以在 Admin 控制台的 Web 界面上直接编写、维护脚本代码。支持 `Java`、`Shell`、`Python`、`NodeJS` 等等。
*   **免部署动态生效**：修改完 GLUE 代码，点击保存，下一次调度**立刻生效改动**！完全不需要重启你的 Spring Boot 或任何业务服务进程。
*   **应用场景**：编写临时且简单的脏数据清理脚本、每日自动跑一个 Shell 脚本来备份服务器日志等。在执行器所在的机器上运行时，就像执行本地脚本一样顺滑。

## 5. `XxlJobHelper.log` vs `log.info`

在 XXL-JOB 的 Handlers 代码里，经常会面临这两种日志组件的选择。

| 日志组件 | 运作方式 | 归属与查看方式 | 适用场景 |
| :--- | :--- | :--- | :--- |
| **`log.info(...)`** <br>(SLF4J/Logback) | 被框架内部捕获，写入应用所在的本地盘。 | 记录在项目的 `server.log` 或纯文本日志文件中。<br>需要通过 SSH 登入服务器或统一监控系统排查。 | **纯技术调试**。例如异常堆栈打印、SQL 打印、单纯的防坑调试输出等。 |
| **`XxlJobHelper.log(...)`** <br>(XXL 核心库) | 写入配置文件 `xxl.job.executor.logpath` 下独立的归档日志内。 | **可通过 XXL-JOB Admin 的 Web 面板实时渲染查看！** 用户不接触服务器即可阅读日志。 | **业务进度播报**。你想暴露在外界看的提示，如：“目前进度 30%”、“抓取异常跳过”、“成功处理 ID=99”。 |

**开发建议：对内调试用 `log`，对外播报用 `XxlJobHelper.log`。**
