# Quartz 触发器 (Trigger) 核心分类与实战选型

Quartz 内部包含多种 Trigger 子类，但在日常的企业级开发中，真正面向程序员使用的核心触发器主要有 4 种。其余的通常是框架内部使用的底层结构。

## 👑 1. CronTrigger（绝对的“王者”）
*   **出场率**：95% 的业务场景
*   **功能**：使用 `Cron 表达式` 来进行时间调度的触发器。如果它认第二，没人敢认第一。
*   **适用场景**：只要你的任务是跟**“自然日历时间”**强相关的复杂逻辑，全用它！
*   **人类语言举例**：
    *   “每个月的最后一天晚上 23:59 执行”
    *   “每周一早上 8:30 执行”
    *   “每天中午 12 点和下午 6 点各执行一次”
*   **代码构建方式**：
    ```java
    CronScheduleBuilder.cronSchedule("0 0 12 * * ?")
    ```

## ⏰ 2. SimpleTrigger（简易的“傻瓜闹钟”）
*   **出场率**：4% 左右
*   **功能**：不关心具体的日期或星期，仅仅是“不管刮风下雨，每隔多长时间执行一次，最多总共执行多少次”。
*   **适用场景**：固定频率的轮询、倒计时、心跳检测、固定次数的自动重试。
*   **人类语言举例**：
    *   “系统启动后，**每隔 10 秒钟跑一次**，永远跑下去。”
    *   “延时 5 分钟后发一条短信，**然后每隔 1 分钟重试一次，总共重试 3 次**就停止。”（遇到这种精准控制重试次数的场景，Cron 表达式很难写，但 SimpleTrigger 的 API 可以一句话搞定）。
*   **代码构建方式**：
    ```java
    SimpleScheduleBuilder.simpleSchedule()
        .withIntervalInSeconds(10)
        .withRepeatCount(3) // 或者使用 .repeatForever() 永远执行
    ```

## 📅 3. CalendarIntervalTrigger（特殊日历间隔）
*   **出场率**：约 1%
*   **核心痛点**：它是为了解决 `SimpleTrigger` 解决不了的“日历数学问题”而诞生的。
*   **特殊场景**：如果产品经理让你写一个“**每隔 5 个月执行一次**”的任务该怎么写？
    *   如果是 `SimpleTrigger`：它只能按固定的毫秒算，可是每个月天数是不一样的（比如有 28 天或者 31 天），你根本算不出一个绝对固定的毫秒间隔！
    *   如果是 `CronTrigger`：除了用逗号去穷举月份，Cron 表达式极其难以表达“从今天起，动态每跨越 5 个月执行一次”这种随当前时间流动的间隔。
*   **功能**：原生地支持以 `Years`, `Months`, `Weeks`, `Days` 等日历单位为间隔。就算遇到平年或闰年，底层的日历算法也能自己算对下一次具体的执行日子。
*   **代码构建方式**：
    ```java
    CalendarIntervalScheduleBuilder.calendarIntervalSchedule()
        .withIntervalInMonths(5)
    ```

## 🏢 4. DailyTimeIntervalTrigger（为“工作时间”而生的超级神器）
*   **核心痛点**：如果产品经理让你写一个“**每周一到周五，上班时间（早9点到晚5点），每隔 15 分钟跑一次**”抓取股票数据的任务。
    *   如果只用 Cron 表达式，你可能会被那繁杂的逻辑闭环逼疯，最终写出一大串极难读懂、后续极难维护的字符串，甚至得分开配置好几个 Trigger 才能覆盖周一到周五的时间段重叠。
*   **功能**：完美解决这类“业务时段重叠调度”的痛点！它允许你通过方法链指定某些特殊**星期几**，并且指定一天中精确的**开始时间段与结束时间段**，最后再指定这中间的**轮询频率**。
*   **代码构建方式**：
    ```java
    DailyTimeIntervalScheduleBuilder.dailyTimeIntervalSchedule()
        .onMondayThroughFriday() // 指定周一到周五
        .startingDailyAt(TimeOfDay.hourAndMinuteOfDay(9, 0)) // 每天 9:00 开始
        .endingDailyAt(TimeOfDay.hourAndMinuteOfDay(17, 0)) // 每天 17:00 结束
        .withIntervalInMinutes(15) // 这期间每 15 分钟执行一次
    ```

## 👻 5. 补充说明：IDE 提示出的 MutableTrigger 和 CoreTrigger 是干嘛的？
*   **结论：千万别去实例化并使用它们！**
*   它们俩属于框架底层保留接口，并不是业务逻辑的概念。
    *   `MutableTrigger`：字面意思是“属性可以在运行时被修改的 Trigger”，一般供 Quartz 内部的持久化机制进行反序列化时设值，或者在任务尚未绑定进调度器之前作为临时对象来调整参数用。
    *   `CoreTrigger`：更是底层调度引擎为了统一以上四种触发器的行为，而抽象出来供核心线程消费的内部接口。
*   你在 IDE 的继承列表或源码依赖树里如果看到它们，只要在脑海里默默打上 `@Internal` 注解将其过滤即可。

---

## 💡 终极选型背诵口诀：
1. 遇到**控制重试次数、后台死循环**等固定且单纯的物理时间间隔：直接选 **`Simple`**。
2. 遇到必须**卡在具体工作日并受限于业务几点钟**的轮频调度：直接选 **`DailyTimeInterval`**。
3. 遇到时间跨度必须精准踩在**月/年**周期的逻辑节点上：直接选 **`CalendarInterval`**。
4. 剩下的一切诸如定点生成报表等日常日历调度：全部用 **`Cron`** 一通百通！
